以下の内容で GitHub Issue の起票を準備してください: $ARGUMENTS

## 手順

### 1. 既存 Issue の確認(必須・起票より先に行う)

```
gh issue list --state all --search "<症状や機能のキーワード>" --limit 20
```

キーワードを変えて 2〜3 回検索し、類似 Issue を探す。結果に応じて:

- **open の類似 Issue がある** → 起票せず、その Issue を提示して
  「重複の可能性がある」と報告する。追記(コメント)で済むなら追記を提案する
- **closed の類似 Issue がある** → **再発(リグレッション)の疑い**として扱う。
  `gh issue view <番号> --comments` で当時の原因と対応を確認し、
  新規 Issue の本文に過去 Issue へのリンクと「再発の疑い」を明記する
- **類似なし** → 新規起票に進む

### 2. 軽い原因分析(バグの場合のみ)

- 関連しそうなファイル・モジュールの当たりを付ける(コードを深追いしない)
- 分析結果は「推定」と明記して本文に含める。断定しない

### 3. 本文の作成と承認

以下の構成でドラフトを作り、**起票前に必ず本文を提示して承認を得る**:

```
## 概要
## 再現手順 / 発生条件 (バグの場合)
## 期待する動作
## 関連情報 (関連ファイルの推定、類似・過去 Issue へのリンク、再発の疑いの有無)
```

### 4. 起票

承認後に `gh issue create --title "..." --body "..."` で起票し、
発行された Issue 番号と URL を報告する。

注意: ラベルやアサインの運用ルールが決まったら、この手順に追記すること。
