# 主要な設計判断の記録

実装時に「なぜこの仕様か」を確認できるよう、経緯を残す。より詳細な検討過程は、本リポジトリの外で別管理している要件定義書（Word）に記録されている（本リポジトリには含めていない）。

## 技術スタック

- **Next.js (App Router) + TypeScript + Tailwind CSS**。フロントエンド・バックエンド両方を単一リポジトリで管理。App Router を採用し、Server Components / Server Actions を活用する方針。
- **Prisma ORM (v7)**。スキーマファイル（`prisma/schema.prisma`）をDBの唯一の真実の源とする。Prisma 6+ の新 config 方式（`prisma.config.ts`）を採用。
- **Auth.js v5（next-auth@beta）+ @auth/prisma-adapter**。SSO 認証のアダプタとして使用。ID/パスワード認証は提供しない方針に合致。
- **Vercel Postgres（Neon、PostgreSQL 16）**。Vercel へのデプロイと同一エコシステム。接続文字列は `DATABASE_URL`（コネクションプーリング用）と `DIRECT_URL`（マイグレーション用直接接続）の2本を使用。

## 命名規則（2026-07 確定）

- **サロゲートキーのPKはカラム名`id`で統一する**。従来は`id`と`<テーブル名>_id`が混在していたが、Prismaの慣習に沿い、FKとの読み分け（`skill.id`を`skill_id`が参照）も明快なため`id`に統一した。実装前のため変更コストなし。**例外はemployeeのみ**: PKは管理職が採番する業務上の社員ID（VARCHAR(6)）であり、サロゲートキーではないため`employee_id`のまま維持する。

## 認証

- **SSO（Azure AD／Google／GitHub）のみを認証経路とする。ID/パスワード認証は提供しない**。当初はテスト・開発環境限定でID/パスワードログインを用意する方針だったが、最終的に廃止した。理由: 運用負荷が大きく、既存の認証基盤（SSO）だけで十分と判断したため。テストは各SSOプロバイダのテスト用アカウントで代替する。
- **新規社員はメールアドレスの事前登録で照合する**。管理職がEDT006で社員ID・メールアドレスだけを先に登録し、本人が初回ログイン時にメールアドレス一致で認証、その場でexternal_id・auth_providerを確定する。理由: SSOプロバイダのゲスト・個人アカウント等が誤って初回登録フローに入り込むのを防ぐため。
- **Google／GitHubアカウントでのログインにも対応する**。`user_account`の識別子カラムは特定プロバイダに依存しない`external_id`（汎用）とし、`auth_provider`で使用プロバイダを記録する。1アカウントにつき初回ログインで使用したプロバイダに固定し、途中でのプロバイダ切替は非対応。**異なるプロバイダで認証された場合はログインさせず「プロバイダ不一致」エラーを表示する**（挙動が未定義だったため実装可能なエラー仕様として確定。文言はAUTH001参照）。GitHubは個人アカウントで会社メールと紐付いていない場合、未登録エラーになる制約がある。

## 組織構造

- **事業部＞部署＞Grの3階層を1つの自己参照テーブル（organization_unit）で表現する**。当初「department」という単一テーブルで検討したが、「事業部のみ所属」「部署までしか所属しない社員」等、所属の深さが人によって異なることが判明し、3つの固定テーブルでは矛盾が生じるため自己参照方式に変更した。

## 退職者の扱い

- **退職判定は`employee.employment_status`（現職／退職）という専用カラムで行う**。当初は`user_account.deleted_at`（論理削除の日付）を退職判定に流用する方針だったが、最終的に専用カラムに変更した。理由: `deleted_at`は本来「レコードの論理削除」という別目的の汎用カラムであり、退職という業務上重要な状態を汎用カラムに混在させると意味が読み取りにくくなるため。`deleted_at`はレコード削除の監査用として残し、退職判定には使わない。
- **EDT007に「現職に戻す」操作を設ける（2026-07確定）**。誤操作による退職処理の取り消しや再入社に、DB作業なしで画面から対応できるようにするため。退職中のアカウントにのみ表示する。

## スキル・資格・マスタ全般

- **スキル名・資格名・現場ポジション名・現場名はシステム全体でユニーク（カテゴリをまたいでも重複不可）**。表記ゆれ・重複データ防止を優先した。一時的にユニーク制約なしの方針を検討したが、各テーブル内で重複を許容したくないという要件により、最終的に元の「システム全体でユニーク」に確定した。
- **使用中（他レコードから参照されている）のマスタは削除不可とする（2026-07確定）**。削除操作時に参照チェックを行い、参照があればエラー表示する。skill_versionの`is_active`（非表示化）は「新規選択を止めたいが過去データは残したい」場合の手段として併存させる。理由: 参照を残したまま削除を許すと経歴書の表示が壊れ、is_active方式への全面統一はマスタ整理（完全な削除）ができなくなるため。
- **カテゴリはスキルと資格で完全に独立したマスタ**。スキル側でカテゴリを追加しても資格側には影響しない。
- **スキルのバージョンは複数同時登録可能、未入力なら「バージョン管理なし」**。当初は「バージョン管理チェックボックス＋単一バージョン入力」だったが、実際には1スキルに複数バージョン（Java 8/11/17等）を登録するニーズがあるため変更した。
- **project_skill・project_role_linkにも複合ユニーク制約を設ける**（employee_skillと同方針）。同一プロジェクトへの同一スキル・同一役割の重複登録をDB制約で防ぐ。親レコードの論理削除時は従属する子レコードも同一トランザクションで論理削除する（schema.md 一般ルール参照）。
- **`employee_skill.skill_version_id` は sentinel 0 ではなく NULL で「バージョン管理なし」を表す**。元仕様では `skill_version_id=0`（無効な FK 値）を番兵値として使う設計だったが、Prisma の FK 制約と相性が悪いため `NULL` 可に変更した。複合ユニーク `(employee_id, skill_id, skill_version_id)` には PostgreSQL 15+ の `UNIQUE NULLS NOT DISTINCT` を適用し、`NULL` 同士も同一視して重複を禁止する。Prisma 7 はこの構文をスキーマでネイティブサポートしないため、`prisma migrate dev` で生成したマイグレーション SQL の `CREATE UNIQUE INDEX` 末尾に `NULLS NOT DISTINCT` を手動で追記する運用とする。

## データ移行

- **既存Excelからの移行・インポート機能は提供しない**。当初は「一括インポート機能を作る」方針だったが、最終的に不要と判断し、新システムでは社員が新規に入力する運用に変更した。

## PWA方針

- オフライン編集・ホーム画面追加導線・プッシュ通知は**いずれも対応不要**。スマートフォンからの更新のしやすさが目的であり、これらの機能は優先度が低いと判断した。

## 画面構成の変遷

- 当初PDF/Excel出力は独立画面（旧OUT001）だったが、マイページ・経歴書詳細に出力ボタンを統合し、「PDF出力ボタン→印刷用プレビュー（REF005）→ダウンロード/印刷」の2段階フローに変更した。
- 現場名（site）は当初自由記述だったが、表記ゆれ防止のためマスタ化した（MST005 現場マスタ管理を新設）。命名は他マスタと同様にシステム全体でユニーク。
- スキルマップ／組織ダッシュボード（REF008）は、組織単位ごとのスキル・資格保有状況を可視化するために追加した機能。

## 経験年数の計算（2026-07 確定）

- **経験年数は全プロジェクト期間の和集合（重複期間は1回）を月数換算し、12で割った整数部とする**。プロジェクトの掛け持ち（期間重複）がある場合、単純合算では実経験年数を超えてしまうため。進行中（`end_date=NULL`）のプロジェクトは計算時点の年月まで含める。再計算のタイミングはプロジェクト経歴の登録・更新・削除時。

## ロールと画面の整合（2026-07 確定）

- **一般社員も同一部署の社員の経歴書を閲覧できる（2026-07 仕様変更）**。当初は「自分の経歴書のみ」だったが、部署内での相互参照（経歴書の書き方の参考、チーム内のスキル把握）を可能にするため開放した。「同一部署」は所属組織単位から遡った部署（unit_level=2）の一致で判定し、事業部直下・未所属の社員は対象外（判定ルールの詳細はscreens.mdのREF002）。編集は従来どおり本人（＋管理職の自分の分）のみ。
- **REF008（スキルマップ）は全ロール閲覧可能（2026-07 仕様変更）**。当初は管理職のみだったが、集計情報は組織全体で共有する価値があるため開放した。ただし保有者名の表示は経歴書の閲覧権限に連動させ、一般社員には同一部署の社員のみ氏名を表示し、それ以外は人数のみとする（閲覧制限がREF008経由で骨抜きになるのを防ぐため）。

- **人事・営業は自分の経歴書を作成しない（マイページなし）**。当初REF001では「マイページタイルは全ロール共通」としていたが、人事・営業の職務は他社員の経歴書の検索・閲覧であり、自身の経歴書登録のニーズはないと判断し、マイページタイルの表示対象を一般社員／管理職に限定した。これに伴い、人事・営業の初回ログインはEDT001を経ずREF001へ直行し、`is_registered`はログイン成立時に自動でTRUEとする。
- **EDT003（スキル登録）／EDT004（資格登録）のカテゴリは既存マスタからの選択のみ**。当初は新規入力候補の表示も許す記述だったが、マスタ管理は管理職専用（MST001/002）という原則と矛盾するため、選択のみに確定した。カテゴリの追加が必要な場合は管理職に依頼する運用とする。
